# Database

Postgres schema + seed for the Panama Tropical Store.

## Layout

```
db/
  migrations/
    0001_init.sql   # creates products table + updated_at trigger
  seed.sql          # inserts the 6 initial products (idempotent)
```

## Schema notes

- `products.id` is a plain `INTEGER` primary key (no sequence). The 6 seeded
  products use explicit ids 1..6 to match what the UI used before. If you start
  inserting more products from the app, switch this to `BIGSERIAL` (or use
  `GENERATED BY DEFAULT AS IDENTITY`) at that point.
- Prices are stored as `INTEGER` (whole USD), because the current UI renders
  whole-dollar amounts. If you ever need cents, change to `NUMERIC(10,2)` and
  update `mapProductRow` in `src/lib/db.ts` accordingly.
- `created_at` / `updated_at` are managed by a `BEFORE UPDATE` trigger so the
  app does not need to set `updated_at` manually.

## Applying the migration + seed

You need a `psql` client that can reach the Postgres service. The connection
string lives in `DATABASE_URL`.

### Option A — From inside the Easypanel network (recommended)

If your Postgres is the Easypanel "numapetstore_database" service, the
hostname `numapetstore_database` is only resolvable from containers in the
same Docker network. Open a shell inside any other container in that network
(or attach to the database container itself) and run:

```bash
export DATABASE_URL='postgres://numapetstore:numapetstore@numapetstore_database:5432/numapetstore?sslmode=disable'

psql "$DATABASE_URL" -f db/migrations/0001_init.sql
psql "$DATABASE_URL" -f db/seed.sql
```

If `psql` isn't already in the container, install it (`apk add postgresql-client`
on Alpine, `apt-get install -y postgresql-client` on Debian/Ubuntu).

### Option B — From the Easypanel web console

Easypanel's Postgres service has a built-in web SQL console. Paste the
contents of `0001_init.sql` first, run it, then paste `seed.sql` and run it.

### Option C — From your local machine (port-forwarded)

`numapetstore_database` will not resolve from your laptop directly. Either:

1. Port-forward the database via Easypanel's "expose" feature (or an SSH
   tunnel) so it's reachable on `localhost:5432`, then run:

   ```powershell
   $env:DATABASE_URL = "postgres://numapetstore:numapetstore@localhost:5432/numapetstore?sslmode=disable"
   psql "$env:DATABASE_URL" -f db/migrations/0001_init.sql
   psql "$env:DATABASE_URL" -f db/seed.sql
   ```

2. Or run the same `psql` commands inside a one-shot Docker container that
   joins the Easypanel network.

## Convenience scripts

`package.json` exposes:

```bash
npm run db:migrate   # psql "$DATABASE_URL" -f db/migrations/0001_init.sql
npm run db:seed      # psql "$DATABASE_URL" -f db/seed.sql
```

These both shell out to a local `psql`. If you don't have `psql` installed
locally, use Option A or Option B instead.

## Cache invalidation

Product reads are wrapped in `unstable_cache` with `tags: ["products"]` and
`revalidate: 60`. After mutating the table out-of-band, you can either wait up
to 60 seconds, or — if you add a server action that mutates products in the
future — call `revalidateTag("products")` from `next/cache` to invalidate the
cache immediately.
